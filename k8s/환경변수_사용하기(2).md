# K8s 환경변수 관리

<br />
<br />

* Secret · ConfigMap · 로컬 개발 환경 설정

---

```
쿠버네티스 환경에서 환경변수는 .env 파일 대신 Secret 또는 ConfigMap으로 관리한다.

코드에서는 항상 process.env로 동일하게 접근하고,
환경에 따라 주입 방법만 다르다.
```

<br />

| 환경     | 방법                | git 커밋          |
|---------|--------------------|-----------------|
| 로컬 dev | `.env.local`       | 올리면 안 됨       |
| k8s 운영 | `Secret → envFrom` | 올리면 안 됨       |
| 템플릿   | `.env.example`      | 올려도 됨 (값 비움) |

<br />
<br />
<br />
<br />

1. 로컬 개발 환경 (dev)

<br />

`.env.local 파일 생성`

```
여기서는 Next를 기준으로 하지만
대부분 프레임워크가 동일하다.

Next.js가 자동으로 읽으며, env는 git에 올리지 않는다.
```

```zsh
# .env.local

INTERNAL_API_URL=http://localhost:3001
JWT_SECRET=dev-jwt-secret
DB_PASSWORD=dev-db-password
```

<br />

`.gitignore 등록`

```zsh
# .gitignore

.env
.env.local
.env.development
.env.production
```

<br />

`.env.example 만들어서 git에 공유`

```
값을 비워둔 템플릿이다.

Next.js가 읽지 않으며,
팀원에게 "이런 환경변수가 필요해"를 알려주는 문서 역할을 한다.
```

```zsh
# .env.example (git에 올려도 됨 - 값은 비워둠)

INTERNAL_API_URL=
JWT_SECRET=
DB_PASSWORD=
```

<br />
<br />

`Next.js 환경변수 파일 우선순위`

| 파일 | 적용 환경 |
|------|-----------|
| `.env.local` | 로컬 dev (git X) |
| `.env.development` | `next dev` 실행 시 |
| `.env.production` | `next build/start` 실행 시 |
| `.env` | 모든 환경 공통 |

2. K8s 운영 환경 (Secret)

<br />

`Secret vs ConfigMap`

| 구분 | Secret | ConfigMap |
|------|--------|-----------|
| 용도 | 민감한 정보 | 일반 설정값 |
| 예시 | JWT_SECRET, DB_PASSWORD | API_URL, PORT |
| 저장 방식 | base64 인코딩 | 평문 |

<br />

`secret.yaml 작성`

```yaml
# secret.yaml (git에 올리면 안 됨)

apiVersion: v1
kind: Secret
metadata:
  name: next-secret
type: Opaque
stringData:
  INTERNAL_API_URL: http://nest-backend:3000
  JWT_SECRET: your-jwt-secret
  DB_PASSWORD: your-db-password
```

```
stringData는 평문으로 작성하면,
k8s가 자동으로 base64 인코딩해서 저장한다.
```

<br />

`Secret 클러스터에 적용`

```zsh
$ kubectl apply -f secret.yaml

# 확인
$ kubectl get secret next-secret
$ kubectl describe secret next-secret
```

<br />

`Deployment에 Secret 주입`

```yaml
# next-deployment.yaml

apiVersion: apps/v1
kind: Deployment
metadata:
  name: next-frontend
spec:
  replicas: 1
  selector:
    matchLabels:
      app: next-frontend
  template:
    metadata:
      labels:
        app: next-frontend
    spec:
      containers:
        - name: next
          image: your-next-image
          envFrom:
            - secretRef:
                name: next-secret  # secret.yaml의 metadata.name과 일치해야 함
```

<br />

`Next.js 코드에서 사용`

```typescript
// 별도 import 없이 그냥 사용
const res = await fetch(`${process.env.INTERNAL_API_URL}/users`)
const secret = process.env.JWT_SECRET
const dbPw = process.env.DB_PASSWORD
```

<br />

```
코드는 로컬 .env.local이든,
k8s Secret이든 항상 process.env으로 동일하게 접근한다.
```

3. 전체 흐름 요약

```
# 로컬 dev
.env.local -> Next.js가 자동으로 읽음 -> process.env

# k8s 운영
secret.yaml -> kubectl apply
  -> k8s Secret 저장
  -> Deployment envFrom으로 Pod에 주입
  -> process.env

# 코드는 동일
process.env.INTERNAL_API_URL  // 어느 환경이든 동일하게 접근
```

<br />

```
Secret은 git에 절대 올리지 않는다.

값은 팀원끼리 별도 채널로 공유하거나 Vault 같은 시크릿 관리 툴을 사용한다.
```
